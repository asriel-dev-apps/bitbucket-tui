# M5 実装仕様書 — PR詳細の閲覧体験強化（フォーカス/スクロール/インライン画像/リンク/マウス）

作成: 2026-07-11。発端は実運用の PR レビュー（長い日本語本文・テーブル・添付画像を含む PR）で見つかった閲覧不能バグと要望。

## ゴール

PR 詳細画面をブラウザ版に近い読書体験にする: 本文を確実に末尾まで読め、画像は概要内にインライン表示され、リンクを開け、マウスでも操作できる。

## 背景（動機となった実バグ・要望）

1. **概要が途中までしか読めない**: 概要ペインは `Paragraph` + `Wrap` で描画しており、ratatui の実装（`vendor/ratatui/src/widgets/paragraph.rs::render_text`）では wrap 有効時のスクロールオフセットは**折り返し後の表示行**で消費される。一方 `App::detail_max_scroll`（`app.rs`）は tui_markdown の**折り返し前の論理行数**で上限を計算するため、折り返しが多い本文（日本語長文=全角2セル、長URL、テーブル）では上限が過小になり末尾に到達できない。tui-markdown はテーブル非対応（`Tag::Table … not yet supported`）でセルが1論理行に線形化されるため、実 PR では乖離が特に大きい。
2. j/k・矢印が変更ファイル一覧に固定割当で、概要・コメントを直感的にスクロールできない（概要スクロールは PgUp/PgDn・Shift+J/K のみ、コメントはスクロール手段なし）。
3. `i` の画像取得が PR 添付画像 URL（`https://bitbucket.org/repo/{id}/images/{file}.png`）で `https://bitbucket.org/account/signin/?next=…` へ 302 され「通信エラー: error following redirect」で失敗する。
4. 本文中のリンクを開く手段がない（`o` は PR ページ自体のみ）。
5. マウス完全非対応。

## スコープ（5.1〜5.5。実装は番号順を推奨、5.5 は独立）

### 5.1 ペインフォーカス + フォーカス先スクロール + スクロール上限修正

- `PullRequestDetail` にフォーカス状態 `DetailFocus { Overview, Files, Comments }` を追加。**初期値 Overview**（詳細を開いた直後は本文を読むため。従来の「j/k=ファイル一覧」から挙動変更になる点はヒントバーで明示）。
- `Tab` / `Shift+Tab`（BackTab）でフォーカス巡回（Overview→Files→Comments→…）。フォーカス中ペインの枠のみ `themed_block(theme, true)`（`border_focus`）。
- キー割当（フォーカス別）:
  - Overview: `j/k/↑↓`=1行、`Shift+J/K`=10行、`PgUp/PgDn`=viewport 行、`g/G`=先頭/末尾
  - Files: `j/k/↑↓`=選択移動（現行）、`Shift+J/K`=10件、`g/G`=先頭/末尾
  - Comments: Overview と同じスクロール系
  - `d/c/a/x/M/o/i/L/Esc/q/?` はフォーカス無関係に従来どおり
- コメントペインにスクロール状態を追加（viewport を描画時に書き戻し + clamp。`DiffState`/`LogView` と同パターン）。
- **スクロール上限の正確化**: Cargo.toml の ratatui に feature `unstable-rendered-line-info` を追加し（vendor 済みクレートのフラグ有効化のみ。依存追加・再 vendor 不要でオフライン完結）、`Paragraph::line_count(width)` = **折り返し後の実表示行数**で概要・コメント両ペインの上限を計算する。`App::detail_body_rendered_lines` の意味を「折り返し後」に変更し、doc コメントも更新。
- ヒントバー（`hint_entries`）・ヘルプモーダルの表記更新（`Tab`=ペイン移動 等）。

### 5.2 概要ペインのリッチドキュメント化（自前折り返し + インライン画像）

目的: ブラウザ同様に本文中の画像を概要内にインライン表示する。あわせて折り返しを自前で確定させ、スクロールとリンク位置を厳密化する（5.4 のクリック判定の土台）。

- 新モジュール `tui/richdoc.rs`（名称は実装者判断）:
  - 本文 Markdown を `![alt](url)`（コードフェンス外。`replace_image_syntax_outside_code_fences` と同じ走査規則）で分割し `Vec<DocBlock>` を作る: `Text(Vec<Line>)`（tui_markdown 描画結果）/ `Image { alt, url }`。
  - テキストブロックはペイン内幅で**自前 greedy wrap**（grapheme 単位、unicode-width。全角=2セル）して視覚行に確定する。Span 境界とスタイルを保ったまま分割すること。
  - **リンク位置表**: wrap 結果から `(視覚行, 列範囲) → URL` の表を作る。tui_markdown はリンク URL を Span に残さないため、**リンク抽出は Markdown ソースから独自に行い**、リンクテキストの視覚位置と対応付ける。厳密な対応付けが過大なら簡略化してよい: 「リンクを含む視覚行」単位のヒットとし、行内リンクが 1 つならそれを開く / 複数ならリンクパレット（5.3）を開く。
  - 画像ブロックの表示高さ: `image_picker` のフォントサイズ（セル縦横 px）と画像の縦横比から、幅=min(画像本来のセル幅, ペイン内幅)、高さ=アスペクト維持で ceil、**上限 20 行**。未ロード・失敗・`image_picker=None` はプレースホルダ 1 行（現行文言ベース。失敗時は `[画像: alt]（取得失敗: o でブラウザ）`）。
- 仮想ドキュメント高さ = Σ(テキスト視覚行 + 画像高さ)。`detail_scroll` はこの高さで clamp（5.1 の `line_count` 方式を概要側だけ置換。コメント側は 5.1 のまま）。
- 描画: ブロックを歩き scroll 分をスキップし、テキストは視覚行を直接描画（`Paragraph::scroll` は使わない）。画像は `StatefulImage` + `Resize::Crop`（上端で欠ける場合 `CropOptions { clip_top: true, .. }`、下端は既定クロップ）で部分描画。画像ごとに `StatefulProtocol` を保持（URL キーのキャッシュ。既存 `App::image_cache` はデコード済み `DynamicImage` の再利用に使う）。
- 画像ロード: PR 詳細表示時に本文中の全画像参照を非同期取得（既存の画像取得コマンド/`get_image_bytes` を流用。上限・キャッシュも既存どおり）。
- **重要な制約（5.5 の調査結果）**: Bitbucket 添付画像（`bitbucket.org/repo/{shortId}/images/*`）は API token では取得不能のため、インライン描画できるのは**取得可能な画像（外部ホストの公開画像等）のみ**。添付画像はプレースホルダ行（「API token では取得できません: クリック/o でブラウザ」）のままとし、5.4 のクリックでブラウザを開けるようにする。
- ImageView（`i`）は拡大表示用として存続。
- 既知の注意: sixel 系端末は文字と画像の重ね描きで乱れることがある（`Resize::Crop` の既知事項。ratatui-image の doc コメント参照）。

### 5.3 リンクパレット（`L`）

- PR 詳細で `L`: 本文とコメント本文から以下を抽出し、出現順で列挙するモーダルを開く:
  - markdown リンク `[text](url)` と裸 URL（`http://`/`https://`）。コードフェンス内は除外。重複 URL は初出のみ。画像 URL（`![alt](url)`）は含めない（`i` がある）。
- 表示は 1 行 1 リンク: `リンクテキスト（無ければ URL）` + 補助でドメインを淡色表示。`j/k/↑↓` 選択、`Enter`=ブラウザで開く（`open_pr_in_browser` の `open` spawn を関数に切り出して共用）、`Esc`=閉じる。
- 0 件なら Status に「本文にリンクがありません」。
- ヒントバー・ヘルプ更新。

### 5.4 マウス対応

- `Tui::init` で `EnableMouseCapture`、`restore()` と panic hook で必ず `DisableMouseCapture`（RAII 復元の対に含める）。
- `event.rs`: `Event::Mouse(MouseEvent)` を `Msg::Mouse` として `update` へ（`Down(Left)` / `ScrollUp` / `ScrollDown` のみ。Move/Drag/他ボタンは無視）。
- ui は毎フレーム、各ペイン・モーダル・ヒントバー項目の `Rect` を App のレイアウト表へ書き戻す（viewport 書き戻しと同パターン）。update 側はそれでヒットテスト。
- **ホイール**: カーソル位置のペインをスクロール（1 ノッチ=3 行）。対象: PR 詳細 3 ペイン・Diff・各一覧（=選択移動）・StepLog・FileView・モーダル内リスト。
- **左クリック**:
  - PR 詳細: クリックしたペインへフォーカス移動。
  - 一覧行: その行を選択。**選択済み行の再クリック = Enter 相当**（リポジトリ→PR 一覧、PR→詳細、リンクパレット→開く 等）。
  - 概要内: リンク位置表（5.2）に当たれば `open`。画像・プレースホルダのクリックは ImageView。5.2 実装前の暫定は「リンクを含む行クリック→行内 1 件なら open / 複数なら L パレット」でよい。
  - ヒントバー項目クリック = 対応キーと同動作。
  - モーダル外クリック = Esc 相当。**確認モーダル（merge 等の破壊的操作）はクリックでは決定しない**（誤クリック防止。決定は Enter のみ）。
- 制約を README とヘルプに明記: mouse capture 中は端末ネイティブのテキスト選択が Shift+ドラッグになる。ターミナルマルチプレクサ（tmux 等）ではパススルー実装に依存する。

### 5.5 画像取得の失敗を明確なエラー + ブラウザ誘導に変える

- 事象: `get_image_bytes`（`api/client.rs`）が PR 添付画像 URL で signin へ 302。reqwest 既定のリダイレクト追従の結果「error following redirect」の生エラーが Status に出る。
- **調査結果（2026-07-11 確定）: PR 説明文/コメントに貼られた添付画像（`https://bitbucket.org/repo/{shortId}/images/{file}` 形式）は、API token では取得できない。Bitbucket Cloud 側の制約であり回避手段は存在しない。**
  - API token が通るのは `api.bitbucket.org`（REST、email:token）と `bitbucket.org` の Git smart-HTTP（`x-bitbucket-api-token-auth`:token）のみ。一般 web ルートは対象外で、公式に「API token では bitbucket.org にログインできない」と明言されている（Atlassian サポート文書 "Using API tokens" / "API tokens"）。
  - REST API 側の代替も無い: PR コメント/説明文内画像の取得 API は非提供と Atlassian が明言（community.atlassian.com 2022-08-25）。機能要望 BCLOUD-22105 は未解決（Gathering Interest、最終更新 2026-06-30）、PR 添付 API の BCLOUD-21023 は Won't Fix。Issue 添付には API があるが PR には無い。既存 OSS CLI（avivsinai/bitbucket-cli、gildas/bitbucket-cli）も PR 添付取得は非実装。
- したがって実装は次に確定:
  1. `get_image_bytes` で **`bitbucket.org/account/signin` へのリダイレクトを検知**し（リダイレクト追従をやめ手動判定 or リダイレクト先 URL 検査）、`ApiError` に専用バリアントを追加。Status には「この画像（Bitbucket 添付）は API token では取得できません。o でブラウザ表示してください」を出す（生の reqwest 文言をやめる）。
  2. ImageView / インライン画像（5.2）のプレースホルダも同じ文言に統一。
  3. **bitbucket.org / api.bitbucket.org 以外のホストへは Authorization を付けない**（LEDGER 記載の「外部ホストへの資格情報送出リスク」の解消。リダイレクト先にも適用）。外部ホストの公開画像は無認証で取得を試みる（成功すれば表示できる）。
- `MAX_IMAGE_BYTES` / 画素上限は現行維持。

## やらないこと（後回し）

- コメントの木構造ナビゲーション、コメント内画像のインライン表示（リンクパレットと `i` の対象には含める）
- OSC 8 ハイパーリンク
- Diff 内テキストのマウス選択サポート
- ターミナルマルチプレクサ側の画像プロトコル/マウスイベントのパススルー対応（本プロダクトの外）

## 品質バー（M0〜M4 と同じ、全て green）

- `rustup run stable cargo fmt --check` / `clippy --all-targets -- -D warnings` / `test`（--offline）
- `unwrap()`/`expect()` はテスト以外禁止。TUI 中に stdout/stderr へ書かない。raw mode / alternate screen / **mouse capture** を RAII + panic hook で必ず復元。
- 依存追加は原則なし（vendor 済みクレートの feature 有効化は可）。新規 crate が必要になったら実装前に相談。

## 受け入れ条件（Acceptance）

1. 折り返し・テーブル・長 URL を含む長い日本語本文の PR で、概要を**末尾まで**スクロールできる（Overview フォーカスで `j`/`G`）。
2. コメントペインも同様にスクロールできる。
3. 画像付き PR で、対応端末では概要内に画像がインライン表示され、スクロールで画像が上下端に部分的にかかっても表示が破綻しない。非対応端末では従来プレースホルダ + `i` 案内のまま。
4. `L` でリンク一覧が開き、`Enter` でブラウザが開く。コードフェンス内 URL・画像 URL は列挙されない。
5. マウスホイールで各ペインがスクロールし、クリックでフォーカス/選択/決定/リンクオープンができる。merge 確認モーダルはクリックでは決定されない。
6. 画像取得が成功する（不可と確定した場合は、明確な日本語エラー + ブラウザ誘導になる）。
7. 既存のキーボード操作は回帰しない（`d/c/a/x/M/o/i` と各画面遷移）。全ユニットテスト green。

## 検証方法

- ユニット: 自前 wrap（全角・結合文字・空行・超長 URL・スタイル境界）/ リンク抽出（フェンス除外・重複・裸 URL）/ 仮想高さと clamp / フォーカス遷移 / ヒットテスト / リンク位置表 / 認証ヘッダの付与条件（ホスト別）。
- 手動: 実端末（iTerm2 等）で実 PR に対し受け入れ条件 1〜7 を確認。マルチプレクサ内では画像はハーフブロック・マウスはパススルー実装依存である旨を LEDGER に記録。
