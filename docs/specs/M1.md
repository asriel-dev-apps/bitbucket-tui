# M1 実装仕様書 — PRレビュー(最優先)

対象: `bitbucket-tui/`。上位方針は `../design.md`、規約は `../../CLAUDE.md`、環境事実は `../LEDGER.md` を必ず読むこと。M0 の上に載せる。

## ゴール

M0 の `RepoSelected` を起点に、**選択リポジトリの PR をレビューできる**ようにする: PR一覧 → 詳細 → diff閲覧 → approve / request-changes / コメント / merge。TUI が Web UI に勝つキーボード駆動レビュー体験を M1 で立ち上げる。

## スコープ(M1でやること)

1. API クライアント拡張(PR系エンドポイント + テキスト取得)。
2. 画面追加: `PullRequests`(一覧) → `PullRequestDetail`(詳細) → `Diff`(差分閲覧)。
3. アクション: approve/unapprove、request-changes/取消、**一般コメント投稿**、**merge**(strategy選択 + close source branch)。破壊的操作は確認モーダル。
4. M0 の `RepoSelected` を廃し、repo 選択後は `PullRequests` へ遷移。

### やらないこと(M2以降 / 後回し)

- パイプライン(M2)、ブランチ/ソース閲覧(M3)。
- **インラインコメント投稿**は M1 では任意(実装するなら stretch)。一覧表示(inline 位置つき)は行うが、投稿は一般コメント優先。
- コメントのスレッド返信・編集・削除。
- PR の新規作成・decline。

## 依存(重要)

**新規クレート追加は禁止**(ネットワーク不可・依存凍結)。M1 は既存依存(reqwest/ratatui/tokio/serde等)だけで実装する。diff の色付けは syntect 等を使わず**ユニファイド diff 行を手動着色**(`+`=緑, `-`=赤, `@@`hunk=シアン, context=既定)。どうしても新クレートが要ると判断したら追加せず報告して止まること。

## API 仕様(追加)

base = `https://api.bitbucket.org/2.0`。`{ws}`=workspace slug, `{repo}`=repo slug, `{id}`=PR id。認証・ページング・エラー変換は M0 の `api` を再利用。

- 一覧: `GET /repositories/{ws}/{repo}/pullrequests?state=OPEN&pagelen=50&sort=-updated_on`
  - `state`: `OPEN`|`MERGED`|`DECLINED`|`SUPERSEDED`(複数指定は state を繰り返す)。既定は OPEN。UI で state 切替できると良い。
- 詳細: `GET /repositories/{ws}/{repo}/pullrequests/{id}`
- 差分(テキスト): `GET /repositories/{ws}/{repo}/pullrequests/{id}/diff` → **`text/plain` のユニファイド diff**(JSON でない → `get_paged` とは別に生テキスト取得メソッドを追加)。
- 差分統計: `GET /repositories/{ws}/{repo}/pullrequests/{id}/diffstat`(ページング; ファイル毎の status/lines_added/lines_removed/old.path/new.path)。ファイル一覧に使う。
- approve: `POST .../pullrequests/{id}/approve` / 取消 `DELETE` 同URL。
- request-changes: `POST .../pullrequests/{id}/request-changes` / 取消 `DELETE` 同URL。
- コメント一覧: `GET .../pullrequests/{id}/comments?pagelen=50`(ページング)。
- コメント作成: `POST .../pullrequests/{id}/comments`、body `{"content":{"raw":"..."}}`(一般)。inline は `"inline":{"path":"file","to":<新ファイル行>,"from":<旧ファイル行>}` を付与(stretch)。
- merge: `POST .../pullrequests/{id}/merge`、body `{"merge_strategy":"merge_commit|squash|fast_forward","message":"...","close_source_branch":<bool>}`。大きな merge は 202 + タスクポーリングになり得る点に留意(まずは同期応答前提、202 は「処理中」と表示)。

> **モデル(serde)**: `PullRequest`{id,title,state,author(User),source/destination(branch: {branch:{name}}, commit),created_on,updated_on,comment_count,task_count,summary?,rendered?,close_source_branch,reviewers?,participants[]}、`Participant`{user,role,approved(bool),state?}、`Comment`{id,content{raw,html?},user,created_on,updated_on,deleted?,inline?{path,from,to},parent?{id}}、`DiffStatEntry`{status,lines_added,lines_removed,old?{path},new?{path}}、`MergeParams`。
> **未検証**: 上記フィールドの正確な有無/名称は**実 API の初回応答で検証**し、LEDGER と本仕様に反映すること(公式スキーマをオフラインで取得できなかったため)。不明フィールドは `#[serde(default)]`/`Option` で耐性を持たせる。

必要スコープ: `pullrequest`(読取), `pullrequest:write`(approve/merge/comment)。

## TUI 仕様(追加)

- `Screen` に `PullRequests`, `PullRequestDetail`, `Diff` を追加(`RepoSelected` は削除し、repo 選択で `PullRequests` をロード)。
- **PullRequests(一覧)**: 各行 = `#id  状態  タイトル  author  更新日  ✔承認数/レビュア数`。`Enter`=詳細、`o/m/d/a` などで state フィルタ切替(OPEN/MERGED/DECLINED/ALL)、`r`=再読込。ページング対応(必要なら「さらに読み込む」)。
- **PullRequestDetail**: 上部にメタ(タイトル/状態/source→destination/author/承認状況/コメント数)、本文(description の raw)、下部に diffstat のファイル一覧。キー: `d`=Diff、`c`=コメント投稿、`a`=approve トグル、`x`=request-changes トグル、`M`=merge(確認モーダル)、`Esc`=一覧へ。
- **Diff**: ユニファイド diff をスクロール表示、上記の色分け。`↑/↓ j/k`・`PgUp/PgDn`・`g/G`。可能なら diffstat のファイル境界へジャンプ(`n/N`)。`Esc`=詳細へ。
- **確認モーダル**: `M`(merge)は必ず確認。strategy 選択(merge_commit/squash/fast_forward)と close source branch トグルを提示 → `Enter`で実行、`Esc`取消。approve/request-changes は即時でよいが結果をステータス行に表示。
- **コメント投稿**: 複数行入力できる簡易エディタ(最低1行入力+送信)。送信中/成功/失敗をステータス表示。
- 破壊的アクション(merge)実行後は PR 状態を再取得して反映。非同期処理は M0 同様 `tokio::spawn`→`Msg`。ローディング/エラーはステータス行。
- 共通キー(M0踏襲): `q`/`Ctrl+C`終了, `Esc`戻る, `?`ヘルプ。ヘルプに M1 キーを追記。

## モジュール変更(目安)

- `api/models.rs`: 上記モデル追加。`api/client.rs`: `list_pull_requests`/`get_pull_request`/`get_pr_diff`(text)/`get_pr_diffstat`/`approve`/`unapprove`/`request_changes`/`unrequest_changes`/`list_comments`/`create_comment`/`merge_pull_request`。生テキスト取得の共通ヘルパを追加。
- `tui/app.rs`: `Screen`/`Msg`/`Command`/`update` 拡張。`tui/ui.rs`: 新画面描画 + diff 着色 + モーダル。`tui/event.rs`: 新 Command の dispatch。必要なら `tui/diff.rs`(diff パース/着色)、`tui/modal.rs` を追加。
- `docs/design.md` の画面遷移は実装に合わせて更新。

## 品質バー(M0 と同じ、全て green)

- `export PATH="$HOME/.rustup/toolchains/stable-aarch64-apple-darwin/bin:$PATH"` の上で
  `cargo fmt --check` / `cargo clippy --all-targets --offline -- -D warnings` / `cargo test --offline`。
- `unwrap`/`expect` はテスト以外禁止。bin=anyhow, api=thiserror。
- ネットワークを叩くテストは `#[ignore]`。通常テストはモック JSON/テキスト: PR/Comment/DiffStat のデシリアライズ、diff パース/着色、merge body の JSON 生成、`update()` の画面遷移とアクション発行。
- TUI 中 stdout/stderr 禁止。端末は RAII+panic hook 復元(M0 の `Tui` を再利用)。
- crossterm 型は `ratatui::crossterm` 経由。

## 受け入れ条件(Acceptance)

- [ ] fmt/clippy(-D warnings)/test すべて green(offline)。
- [ ] repo 選択 → PR一覧が出る(OPEN 既定、state 切替可)。
- [ ] PR選択 → 詳細(メタ/本文/ファイル一覧)表示。
- [ ] 詳細から Diff を色付きスクロール表示。
- [ ] approve/unapprove、request-changes/取消 が動き、状態がUIに反映。
- [ ] 一般コメント投稿ができ、一覧に反映。
- [ ] merge が確認モーダル経由で実行でき、strategy と close source branch を選べる。実行後 PR 状態が更新。
- [ ] 破壊的操作(merge)は確認なしに実行されない。
- [ ] エラー(401/403/409 等)がステータス行に分かりやすく表示。

## 検証方法

- ユニット: 上記モデル/diff/merge-body/update をモックで。ネット依存は `#[ignore]`。
- 手動(実 token 必要・任意): テスト用 PR で approve→diff→comment→merge を一巡。手順を README に追記。
- 実 API で判明したフィールド/挙動(state 値・merge 202・inline 位置の意味等)を `docs/LEDGER.md` に反映。
