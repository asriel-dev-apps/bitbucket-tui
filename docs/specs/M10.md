# M10: 認証情報保存の刷新（更新耐性 + Linux 対応）

## 背景・問題

現行の token 保存は `keyring` crate（`apple-native`）による macOS Keychain 直接アクセス
（Security.framework を静的リンク）。これには 2 つの実害がある。

1. **更新のたびに再入力**: macOS Keychain の項目 ACL は「項目を書いたプロセスのコード署名」に
   紐づく。`cargo install --force` での更新はバイナリを再ビルドし ad-hoc 署名のハッシュが毎回
   変わるため、既存項目の読み出しが拒否され `restore_client`（main.rs:117-133）が Onboarding に
   フォールバックする。Security.framework 直接リンク型ツールの既知問題
   （docker-credential-helpers #319 / aws-vault #1179 と同型）。
2. **Linux 非対応**: `apple-native` のみ有効のため Linux では mock（非永続）。

**方針（gh CLI / zalando/go-keyring と同方式）**: OS セキュアストアへのアクセスを「署名が安定した
仲介」経由に変える。平文ファイルへの保存・フォールバックは作らない。

- macOS: `/usr/bin/security` サブプロセス。項目の所有者が Apple 署名済みの `security` になるため、
  本体バイナリを何度再ビルドしても ACL が壊れない。
- Linux: libsecret（Secret Service）を **実行時 dlopen**（cargo 公式 `cargo-credential-libsecret` と
  同方式。libloading 使用）。Secret Service にはバイナリ署名に紐づく ACL が無い。
- headless / その他 OS: 環境変数 `BBTUI_EMAIL` / `BBTUI_TOKEN`（永続化しない）。

## 事前準備済み（2026-07-19）

- `Cargo.toml` に `libloading = "0.8"` 追加済み・vendor 再生成済み（`vendor/libloading` あり、
  Cargo.lock 更新済み）。**通常依存にしてあるのは意図的**（libsecret バックエンドを全 OS で
  コンパイル対象にし、macOS 上の check/clippy/test で Linux コードも検証するため）。
- 依存凍結は継続: `cargo add` / `update` / `vendor` 実行禁止（LEDGER.md 参照）。`keyring` の削除は
  本 M の作業に含む（削除だけなら `--offline` で lock が更新できる）。vendor/ 配下の不要クレートの
  掃除は検証後に行うため**触らない**。

## 10.1 `src/auth.rs` の置き換え

- 公開 API は現行の 3 関数を維持: `save_token(email, token)` / `load_token(email) -> Option<String>` /
  `delete_token(email)`。エラー型は `keyring::Error` を廃し、`thiserror` の `AuthError` を新設
  （lib 的モジュールは thiserror の規約どおり）。呼び出し側（main.rs:66-75 `logout` /
  main.rs:117-133 `restore_client` / tui/app.rs:3746-3778 `on_auth_validated`）を型追従させる。
  挙動（失敗時に warn ログ + Onboarding へ、logout の not-found 成功扱い等）は変えない。
- `use keyring::` を全廃し、`Cargo.toml` から `keyring` を削除する。
- モジュール構成: バックエンド 2 つは **cfg で塞がず全 OS でコンパイルされる**下位モジュールにし、
  公開 3 関数のディスパッチだけを cfg 分岐にする:
  - `security_cli`（macOS 用実体。ただしコンパイルは全 OS）
  - `libsecret`（Linux 用実体。ただしコンパイルは全 OS）
  - `#[cfg(target_os = "macos")]` → security_cli、`#[cfg(target_os = "linux")]` → libsecret、
    それ以外の OS → 常に `Err(AuthError::UnsupportedPlatform)`（Windows は本 M のスコープ外）。
- サービス名は現行同様 `"bitbucket-tui"`、アカウント = email。
- モジュール doc コメントは全面書き換え（現行 auth.rs:6-10 は「mock フォールバック」という
  古い記述で実態と不一致。LEDGER.md:32 が正）。新コメントには「なぜサブプロセス / dlopen なのか」
  （署名 ACL 問題）を短く書く。

### security_cli（macOS: `/usr/bin/security` サブプロセス）

- パスは `/usr/bin/security` 固定（PATH 検索しない）。
- **save**: 「delete → add」の 2 段構え。
  1. `delete-generic-password -s bitbucket-tui -a <email>` を実行し、結果は無視する
     （旧 keyring 実装が作った「本体バイナリ所有」の既存項目の除去。無ければ失敗するが無視）。
  2. `/usr/bin/security -i` を起動し、**stdin へ** `add-generic-password -U -s "bitbucket-tui"
     -a "<email>" -w "<token>"` を 1 行書いて閉じる。**token を argv に載せない**（ps 露出防止。
     go-keyring と同方式）。
  - 引用規則: 値は `"…"` で囲み、`\` と `"` をこの順でバックスラッシュエスケープする純関数
    `quote_security_arg` を切り出し、単体テストで固定する（空文字・引用符・バックスラッシュ・
    空白・日本語・`@`）。
- **load**: `find-generic-password -s bitbucket-tui -a <email> -w` を実行（こちらは argv で可。
  秘密は出力側）。exit 0 → stdout の末尾改行を除去して `Some`。**exit 44（errSecItemNotFound）→
  `Ok(None)`**。それ以外 → stderr の要約付き `Err`。
- **delete**: `delete-generic-password -s -a`。not-found（44）も成功扱い（現行と同じ）。
- exit code → 結果の写像は純関数に切り出して単体テストする（サブプロセスをモックしない。
  実行部は薄く保つ）。
- **実 Keychain roundtrip テストを `#[ignore]` で 1 本**用意する（既存のネットワーク smoke と同じ
  扱い。検証時に実機で回す）: service は **`bitbucket-tui-test`** を使い、特殊文字入り token
  （`"`, `\`, 空白, 日本語を含む）で save → load 一致 → delete → load None を確認して掃除まで行う。
  ignore でないテストから実 Keychain に触ることは禁止。

### libsecret（Linux: dlopen）

- 参考実装: cargo 公式 `cargo-credential-libsecret`（単一ファイル・libloading）。同じ流儀で
  `libsecret-1.so.0` を dlopen し、`secret_password_store_sync` / `secret_password_lookup_sync` /
  `secret_password_clear_sync` を解決する。
- schema は attributes `{"service", "user"}` の 2 つで自前定義（`SECRET_SCHEMA_NONE` 相当、
  collection は DEFAULT）。label は `bitbucket-tui (<email>)`。lookup/clear は同 attributes で照合。
- dlopen 失敗（libsecret 未導入）・シンボル解決失敗・GError 返却（DBus 不通 = headless 等）は、
  ユーザー向けに何をすべきか分かる `Err`（例: 「libsecret が見つかりません。デスクトップ環境では
  libsecret を導入、headless では BBTUI_EMAIL / BBTUI_TOKEN を使ってください」）。呼び出し側は
  現行どおり warn ログ + Onboarding。
- GError は g_error_free で必ず解放。`unsafe` ブロックには安全性コメントを付ける。返却された
  password 文字列は `secret_password_free` で解放する（cargo 実装に忠実に）。
- FFI 宣言・エラー写像など**プロセス外に出ない純粋部分は通常テスト**、実 Secret Service に触る
  経路はテストしない（実行環境が無い。検証時に Docker で compile / 可能なら roundtrip を確認）。

### 環境変数（全 OS・最優先）

- `BBTUI_EMAIL` と `BBTUI_TOKEN` が**両方**設定されていれば、`restore_client`（main.rs）は
  config / セキュアストアを読まずにこの値でクライアントを構築する。**保存はしない**（Onboarding
  が出ないので save 経路に乗らないことを確認する）。
- 片方だけ設定されている場合は warn ログを出して通常フローへ。
- `logout` は環境変数に関与しない（保存物のみ消す。環境変数使用中なら「環境変数は消せない」旨を
  println で一言添える）。
- 命名は既存の `BBTUI_LOG` / `BBTUI_TEST_*` と同系。

## 不変条件

- 新規依存の追加禁止（libloading は追加済み。`cargo add`/`update`/`vendor` 実行禁止）。
- テスト外の `unwrap()` / `expect()` 禁止。
- token を平文ファイル・ログ・stdout・tracing に一切出さない（email は現行同様可）。
- TUI の画面・キー操作は変更しない（Onboarding の見た目も不変）。
- `#[ignore]` なしのテストから実 Keychain / 実 Secret Service に触らない。
- ネットワーク禁止（`--offline`）。**commit / push 禁止**（検証完了まで）。
- 壊してはいけない既存機能: `BitbucketClient::Debug` の token redact、`--reset-auth` / `logout` の
  挙動、`BBTUI_TEST_EMAIL` / `BBTUI_TEST_TOKEN` の統合テスト経路（client.rs:1672-1679。今回の
  環境変数とは別物。名前も変えない）。
