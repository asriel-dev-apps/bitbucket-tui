# M9: 実運用フィードバック第 4 陣（キーヒント順序・フィルタ複数選択・ソートラベル・コメント間隔）

実運用フィードバック第 4 陣。Phase 1（小粒 3 件: キーヒント順序・ソートラベル英語化・スレッド内コメント間隔）と Phase 2（Author/Target フィルタの複数選択化）の 2 段で実装する。

## Phase 1

### 9.1 キーヒントの並び順反転（`?` を末尾へ・`Enter` を先頭へ）

**問題**: 利用頻度の低い `?`（ヘルプ）がフッター先頭を占有し、`Enter` 等の主要キーが後方にある（M8 8.1 の「`?` 先頭」方針の反転。ユーザー指示 2026-07-19）。

**仕様**:
- `hint_entries`（ui.rs:2748）の並び方針を「**`Enter`（主要アクション。存在する画面のみ）→ 画面固有キー（現行の相対順を維持）→ `Esc` → `Ctrl+K` → `q` → `?`（末尾）**」へ変更する。
  - 各画面 arm 内で `Enter` エントリがあれば先頭へ移動。それ以外の相対順は変更しない。
  - 共通処理（ui.rs:2877-2881）を「`Ctrl+K`・`q` を push した後、最後に `?` を push」へ変更。
  - doc コメント（ui.rs:2739-2747）の方針記述も新方針に書き換える。
- Onboarding は不変（`?`/`Ctrl+K`/`q` 無し）。ヘルプ全文（`render_help`）は変更しない。
- `render_hints`（ui.rs:2886）: 幅切詰めは現行どおり「丸ごと収まらないエントリで break（部分描画なし）」だが、**末尾の `?` は常に表示**する。末尾 `?` エントリの幅を先に予約し、残り幅に先頭からエントリを詰め、最後に `?` を描画する（右端固定までは不要。最後に必ず入ればよい）。ヒットボックスは描画位置と一致させる。`?` 1 個すら入らない極小幅では `?` のみ表示。
- テスト更新:
  - `hint_entries_put_help_key_first_on_every_screen_except_onboarding`（ui.rs:5905）→「`?` が**末尾**・重複なし」の検証へ書き換え。
  - `render_hints_keeps_help_key_even_when_width_is_very_narrow`（ui.rs:5983）→「狭い幅でも `?` が必ず表示される」検証を維持（位置は末尾）。
  - その他の不変条件（PgUp/PgDn・Shift+J/K 非表示、PR 一覧は `f` のみ等）は維持。
- 新規テスト: `Enter` エントリを持つ全画面で `Enter` が先頭であること。

**問題**: 「更新が新しい順」等の「〇〇順」表記を英語ラベルにしたい（ユーザー指示）。

  - `LeastRecentlyUpdated` → `"Least Recently Updated"`
  - `Newest` → `"Newest"`

- 新規テスト: 4 ラベルの文字列を固定する。

### 9.3 スレッド内のコメント間隔

**問題**: 同一スレッド内で、コメントのアクション行（`Reply · …`）と次コメントのヘッダ行が密着していて読みにくい（M7 1.1 はスレッド**間**の空行のみ。スレッド**内**は未対応）。

**仕様**:
- `push_thread_rows`（app.rs:1092-1138）の `for comment in &thread.comments` ループで、**最後以外の各コメントの `Actions` 行の直後に枠内空行を 1 行挿入**する。
- 新 variant（例: `CommentRowKind::InnerSpacer`。命名は任意）を追加する。スレッド枠の**内側**なので、描画（`comment_box_line` ui.rs:1516-1637）は枠線付き空行（`│` + 空白 + `│`。split は左カラムも同様）。既存の `Spacer`（枠外・`Line::raw("")` ui.rs:1635）とは別物で、`Spacer` の描画は変更しない。
- non-focusable（カーソル素通り）・`click_body_row` で明示 no-op（`Spacer` と同じ扱い）。
- スレッド末尾（最終コメントの `Actions` と `Bottom` の間）には入れない。collapsed 行（`push_collapsed_row` app.rs:1064-1081）は対象外。
- Actions 行のヒットボックス座標は viewport 基準の描画ループに従うため挿入でズレない想定（M7 1.1 と同じ理由）だが、テストで固定する。
- 新規テスト: 複数コメント（root + reply 2 件以上）のスレッドで rows が `Top / Header,Body*,Actions / InnerSpacer / Header,Body*,Actions / Bottom / Spacer` の構成になること、InnerSpacer が non-focusable であること。既存 `display_rows_*` は単一コメント fixture のため原則不変。

## Phase 2: Author/Target フィルタの複数選択化（Phase 1 完了後に着手）

**ゴール**: フィルタモーダルの Author / Target branch セクションを、States と同じ「**チェックボックス複数選択 + 作業コピー**」モデルへ統一する。Space で押した瞬間にチェックが見え、Enter で適用、Esc で破棄。`All` は両セクションとも常時 row 0。選択済み項目は `All` の直下に pin される。

現行は単一選択（カーソル位置＝選択のラジオ式、ui.rs:3201-3205, 3229-3233）で、Space は検索クエリへの文字入力に食われる（app.rs:4994-5005）。モーダル下部の「Space: 切替」（ui.rs:3160-3166）が Author/Target では嘘になっている——これが「Space を押してもチェックがつかない」報告の正体。

### API（事前実測済み 2026-07-19。公開リポジトリ atlassian/aui・無認証）

- `q=state IN (...) AND (author.uuid="{a}" OR author.uuid="{b}")` → 200。指定 2 名の PR のみ返る（和集合）。
- `q=state IN (...) AND (destination.branch.name = "master" OR destination.branch.name ~ "release")` → 200。完全一致と部分一致の和集合が返る。
- 既存規則を踏襲: author/target が有効なとき `q=` 一本化（`state` パラメータ禁止。M7 実測）、文字列は `bbql_quote` でエスケープ、`~` は大文字小文字無視の部分一致（M8 実測）。
- **未検証の仮定**: 認証付きリポジトリでの同挙動（従来どおりユーザーの実利用確認に委ねる）。

### フィルタ状態（適用側）

- `PrStateFilter.author: Option<PrAuthor>` → **`authors: Vec<PrAuthor>`**（空 = All）。uuid で重複排除し、表示名 case-insensitive 昇順（同名時は uuid 昇順）へ正規化して保持する（`Eq`/`Hash` がキャッシュキーのため順序を正規化）。
- `PrStateFilter.target_branch: Option<TargetBranch>` → **`target_branches: Vec<TargetBranch>`**（空 = All）。`(text, exact)` で重複排除・text 昇順へ正規化。
- `bbql_filter`（client.rs:863-888）: 各次元 1 件なら現行どおり、複数件なら `(x OR y OR ...)` と括弧で括って AND 連結。単数アクセサ（`author_uuid()` 等）は複数対応に置き換える。
- `label()`（app.rs:305-319）: 複数は `", "` 連結。部分一致（`exact=false`）は `~` を前置（例: `author: Alice, Bob`・`target: main, ~release`）。
- repo 切替（`select_repo`/`jump_to_pr`）でのリセット・セッション内のみ（永続化しない）は不変。

### モーダル状態（作業コピー）

- `PrFilterModal` に pending 選択集合を追加: `author_selected`（PrAuthor 集合）・`target_selected`（TargetBranch 集合）。`open_pr_filter_modal` で適用中フィルタから初期化する。
- 行モデル `author_row`/`target_row`（app.rs:423-463）を更新。**行→意味の写像はこの 2 関数への一元化を維持**する（描画と適用のずれを構造的に防ぐ。M8 レビューの設計）:
  - **両セクションとも row 0 = `All` を常時表示**（クエリ非空でも・候補ローディング中でも）。M8 の「author はクエリ非空で All 非表示」「branch ローディング中は All 非表示」は、いずれも「Enter=カーソル行適用」前提の不一致回避だったため、本モデル（Enter=pending 適用）では撤廃する。
  - クエリ空: `All` → **選択済み pin 行**（author: 表示名昇順 / target: text 昇順。substring 選択（`exact=false`）も `~ "text"` 表示の pin 行として出す）→ 未選択候補（author: 表示名昇順=現行 / target: API 順=最新コミット順の現行）。
  - クエリ非空: `All` → `Partial`（target のみ。現クエリの部分一致行）→ fuzzy 一致候補（pin なし。チェック表示のみ。選択済みでも一致しなければ出ない——チェック状態は保持される）。
  - **pin の再計算は「モーダル open 時」と「クエリ refilter 時（空へ戻った時含む）」のみ**。Space トグルでは行順を再計算しない（トグルの瞬間に行が動いてカーソルを見失うのを防ぐ）。
- 旧補助関数（`insert_current_author`/`author_cursor_for`/`insert_current_target`/`target_cursor_for`、app.rs:470-522）は pin 方式に置き換え。open 時に「substring フィルタをクエリへ復元」する挙動（app.rs:4852-4872）は廃止する（pending の pin 行として現れるため）。
- カーソル初期位置: 両セクションとも row 0（`All`）。

### キー操作（`on_key_pr_filter_modal` app.rs:4913-5009）

- **Space（Author/Target セクション）: カーソル行をトグル**。
  - `All` 行: pending 集合を空にする（= `All` がチェック状態になる）。既に空なら no-op。
  - 候補行: pending への追加/削除（author は uuid 同定、target は `{text, exact: true}`）。
  - `Partial` 行（target・クエリ非空）: `{text: 現クエリ, exact: false}` をトグル。トグル後にクエリを編集しても、追加済み pending はスナップショットとして変化しない。`Partial` 行のチェック表示は「`{現クエリ, exact:false}` ∈ pending」で判定。
  - ローディング/0 件プレースホルダ行: no-op。
  - **Space はクエリへ入力しない**（printable char ハンドラ app.rs:4994-5005 から `' '` を除外）。fuzzy は部分列一致で空白なしでも複単語名（例: `taroyam` → `Taro Yamada`）に一致するため検索性は損なわれない。
- ↑↓: カーソル移動のみ（現行どおり）。Tab/BackTab のセクション巡回・Esc の全破棄（app.rs:4918-4922）も現行どおり。
- **Enter: モーダル全体を適用** — states（現行維持。空集合拒否も現行維持）+ `authors ← author_selected` + `target_branches ← target_selected`。`apply_pr_filter_modal`（app.rs:5085-5118）のカーソル行参照（All/Candidate/Partial/Missing 分岐）は廃止。pending が空なら All（フィルタ無し）として適用する。

### 描画（`render_pr_filter_modal` ui.rs:3098-3253）

- Author/Target の候補行: ラジオ `(x)/( )` → **チェックボックス `[x]/[ ]`**（pending 集合の membership で判定）。カーソル行は States セクションと同じ視覚方式で示す（チェックとカーソルは独立）。
- `All` 行: `[x]` ⇔ pending が空。
- モーダル下部ヒント「Space: 切替」（ui.rs:3160-3166）は全セクションで真になるため文言維持。
- 候補窓のスクロール・高さ予算の導出（M8 レビュー(3)(6)）は既存ロジックを維持。

- キャッシュキー（`PrStateFilter` の `Eq`/`Hash`）は正規化済み Vec で安定すること（同一選択内容 → 同一キー）。
- 既存テスト `pr_filter_modal_*` / `render_pr_filter_modal_*` は新モデルへ書き換え。
- 新規テスト（最低限）:
  - Space トグルで pending が変わり checkbox が即時反映されること・クエリに空白が入らないこと。
