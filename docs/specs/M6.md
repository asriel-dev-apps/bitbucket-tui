# M6: Diff レビュー体験の強化（インラインコメント表示・返信 / ファイルツリーサイドバー）

PR レビューの中心である Diff 画面を、一般的なコードレビュー UI に寄せる。

- 機能A: Diff 本文の該当行にインラインコメントのスレッドを表示し、その場で返信できる。
- 機能B: ファイル一覧サイドバーを「フォルダ/ファイルの階層ツリー + 追加/削除行数 + コメント数」表示にする。

いずれも新規依存は追加しない。書込み系（返信投稿）は実装するがオフライン検証のみ（実 API へは投稿しない）。

## スコープと前提

- インラインスレッド表示は **unified 表示（`v` で切替）** のみ。split 表示は従来どおり（スレッド非表示）。参照画像も unified。
- スレッドは PR 差分（`diff_return == PullRequestDetail`）でのみ意味を持つ。コミット差分はコメント非対応（従来どおり）。
- コメントは PR 詳細を開いた時点で `App.comments` に取得済み（`LoadComments`）。Diff 画面はこれを参照するのみで追加取得しない。返信投稿後は既存の `refresh_comments()` 導線で再取得される。

## 機能A: インラインコメントのスレッド表示

### データマッピング

コメント（`Comment.inline{path, to, from}`）を unified 行インデックスへ対応づける。

- 対象ファイル: `parsed.files` の `name == inline.path` を検索。
- 行: そのファイル範囲 `lines[file.start..file.end]` から、`inline.to` があれば `new_no == to`、無ければ `inline.from` で `old_no == from` の行を探す（投稿時の `comment_anchor` 規則の逆写像）。
- 行が特定できない（古い差分・範囲外）場合は、そのファイルの先頭行（`file.start`）へフォールバックして表示（情報を落とさない）。

### スレッド化

- ルート = `parent` が `None` かつ `inline` を持つコメント。
- 返信 = `parent` が `Some`。親チェーンを辿ってルートへ集約し、`created_on` 昇順で 1 段インデント表示（PR 詳細ペインの既存表現に合わせる）。
- ルートが一般コメント（inline 無し）のスレッドは Diff には出さない（PR 詳細ペインで見る）。

### 描画（interleave）

`DiffState` に描画非依存の中間データ `comment_layout` を持たせる（`build_comment_layout(parsed, comments)` で構築、`DiffLoaded`/`CommentsLoaded` 時に再構築）。

- `by_line: BTreeMap<usize, Vec<CommentLine>>` — 各 unified 行の直下に差し込む整形済みテキスト行（スタイル種別付き。色はテーマ非依存で、描画時に解決）。
- `reply_targets: BTreeMap<usize, u64>` — 各行の代表スレッドのルートコメント id（`r` 返信先）。
- `file_comment_counts: Vec<usize>` — ファイルごとの inline コメント数（サイドバーのバッジ用。`inline.path` 一致で計数）。

`render_diff_body`（unified）は `scroll` からビューポート高さぶんの **視覚行** を、`diff 行 → その行のコメント行` の順に積んで描画する。split（`render_diff_body_split`）は変更しない。

### スクロールモデルの不変条件（重要）

`scroll`/`cursor` は従来どおり **unified 行インデックス（`parsed.lines` の添字）** のまま維持する。コメント行はカーソル対象にしない。これにより `jump_to_file`/`comment_anchor`/`n`/`N`/`DiffFile.start` など既存のインデックス前提ロジックを一切壊さない。

コメント行は「視覚行数」だけを増やすので、可視性計算のみ拡張する。

- `extra_rows(i)` = unified かつ `by_line[i]` の行数（それ以外 0）。
- `max_scroll` / `ensure_cursor_visible` は「1 行 = 1 視覚行」ではなく `1 + extra_rows(i)` で数える。ビューポート末尾で `cursor` の diff 行が隠れないよう `scroll` を最小限進める。extra_rows が全 0（split・コメント無し）のとき従来式に一致することをテストで固定する。
- `ui.rs` 側の scroll クランプは `DiffState::max_scroll()`（pub 化）へ集約し、二重計算をやめる。

### 返信（reply）

- Diff 本文でカーソル行にスレッドがあるとき `r` で返信エディタを開く（無ければ Status にエラー）。
- `CommentEditor` に `reply_to: Option<u64>` を追加。`reply(root_id)` は `inline=None, reply_to=Some`。エディタ UI は共通、タイトルのみ「返信」。
- `submit_comment`: `reply_to` が `Some` → `Command::CreateReply{parent_id, raw}`。
- API: `create_reply` は `POST .../comments` に body `{"content":{"raw":".."},"parent":{"id":<id>}}`（inline は親から継承。Bitbucket の解釈は未検証の仮定として LEDGER へ）。既存 `create_inline_comment`（新規スレッド）は変更しない。
- 投稿成功 → `Msg::CommentPosted` → `refresh_comments()` → 再取得で `comment_layout` 再構築 → スレッドに返信が反映。

## 機能B: ファイルツリーサイドバー

`parsed.files`（フルパス）から階層ツリーを組み、フラットな表示行列に落として描画する。

- 純関数 `build_sidebar_rows(files) -> Vec<SidebarRow>` を `diff.rs` に置き、単体テストで固定する。
  - `SidebarRow::Folder{depth, name}` / `File{depth, file_index, name(=basename)}`。
  - パスでソートしてフォルダ単位にグループ化。単一子フォルダの連鎖は圧縮して 1 行に（例 `lib/src`）。
  - `/` を含まないファイル・`(unknown)` は深さ 0 のファイル行として扱う。
- 追加/削除行数はファイル範囲の `Added`/`Removed` 行を数えて算出（API 非依存。コミット差分でも動く）。`DiffFile` に `added`/`removed` を持たせ `parse` 時に確定。
- コメント数バッジは `comment_layout.file_comment_counts[file_index]`。
- 描画: フォルダ行は淡色 + `name/`、ファイル行は `basename  +A -R  🗨N`（N>0 のみ）。幅は従来のクランプ（12〜70%）を踏襲し、溢れは末尾優先で省略。

### ナビゲーションの対応付け

`file_index` は従来どおり `parsed.files` の添字（本文ジャンプの `start` はここから引く）。表示順（ソート済みツリー）と添字順が異なるため、リスト系ヘルパを **表示行インデックス空間** で一貫させる。

- `list_len(DiffFiles)` = 表示行数、`list_selection` = 選択ファイルの表示行インデックス、`select_list_position(pos)` = `sidebar_rows[pos]` が File なら選択・Folder なら無視。
- サイドバー ↑↓（`select_file_next/prev`）は表示順の次/前のファイルへ。`n`/`N`（本文の境界ジャンプ）は従来どおり本文（parsed）順。
- `ui.rs` の `first_visible` は選択ファイルの表示行インデックス基準に直す。

## 品質ゲート

- `cargo fmt --check` / `cargo clippy --all-targets -- -D warnings` / `cargo test`（オフライン）を green に保つ。
- 新規テスト: 行マッピング（To/From・フォールバック）、スレッド集約、`extra_rows` を含む `max_scroll`/`ensure_cursor_visible`（境界: コメント無しで従来式に一致）、`build_sidebar_rows`（圧縮・ソート・深さ・unknown）、`create_reply` の body、返信の Command 分岐。
