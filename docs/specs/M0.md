# M0 実装仕様書 — 基盤

対象: `bitbucket-tui/`。上位方針は `../design.md`、規約は `../../CLAUDE.md` を必ず読むこと。

## ゴール

初回起動で認証情報を登録し、ワークスペース→リポジトリを選択して「選択済み画面(M1のPR一覧が入る継ぎ目)」まで到達できる、動作する TUI 基盤を作る。M1以降の土台となるモジュール境界を確立する。

## スコープ(M0でやること)

1. Cargo プロジェクト初期化(単一バイナリクレート、edition 2024、license = "MIT OR Apache-2.0")。
2. 認証情報の保存/読込(keyring = OS Keychain)。
3. Bitbucket API クライアント(reqwest / Basic認証 / ページング / エラー型)と最小モデル。
4. TUI スケルトン(ratatui + crossterm + tokio、mpsc メッセージパターン、端末RAII+panic hook)。
5. 画面: Onboarding → Workspaces → Repositories → RepoSelected(プレースホルダ)。

### やらないこと(M1以降)

- PR / diff / パイプライン / ブランチ等の機能。RepoSelected は "M1でPR一覧" と表示するだけ。

## モジュール構成(目安)

```
src/
  main.rs        // clap CLI, tokio runtime, 端末ガード, panic hook, run()
  config.rs      // 設定ファイル(directories + toml): email, default_workspace
  auth.rs        // Credentials: keyring で token 保存/読込/削除
  api/
    mod.rs
    client.rs    // reqwest Client, Basic認証, get_paged(), エラー変換
    models.rs    // User, Workspace, Repository, Paginated<T> (serde)
    error.rs     // ApiError (thiserror): Auth(401) / Forbidden(403,scope) / RateLimited(429,retry) / Http / Decode
  tui/
    mod.rs
    app.rs       // App状態, Screen enum, update()
    event.rs     // イベントループ(crossterm EventStream + mpsc<Msg>), tokio::select!
    ui.rs        // 各画面の描画
    onboarding.rs// 初回認証入力(email → token マスク入力)
```

lib/bin 分割は任意。単一 bin でも可だが、`api`/`auth` は将来 crate 分離しやすい形に保つ。

- `serde`(`derive`) / `serde_json`
- `keyring`
- `directories`
- `toml`
- `anyhow`(bin) / `thiserror`(api)
- `tracing` / `tracing-subscriber`(`env-filter`)
- `clap`(`derive`)

## 認証 仕様

- Keychain: keyring `Entry::new("bitbucket-tui", &email)` に **API token** を保存(email は config.toml に平文可、token は Keychain のみ)。
- 初回フロー(Onboarding):
  1. email 入力 → token 入力(**画面上はマスク表示**)
  2. `GET /2.0/user` で検証。
     - 200 → Keychain 保存 + config に email/表示名保存 → Workspaces へ
     - 401 → 「メールまたはトークンが不正」。必要スコープ(`account` 等)を明記して再入力
     - 403 → スコープ不足の可能性を明記
  3. 画面下部に「API token は Atlassianアカウント設定 > Security で発行。必要スコープ: account, repository, pullrequest, pullrequest:write, pipeline」を常時ヒント表示。
- CLI: `bitbucket-tui logout`(または `--reset-auth`)で Keychain と config をクリアできること。

## API クライアント 仕様

- base = `https://api.bitbucket.org/2.0`、`Authorization: Basic base64(email:token)`。
- `User-Agent: bitbucket-tui/<version>`。
- `get_paged<T>(path, query)`: `values` を集約し `next` を辿る。**安全上限 20 ページ**(超過時は打ち切り、`log` に記録)。
- エラー変換: 401→`Auth`、403→`Forbidden`(body の scope情報があれば添える)、429→`RateLimited`(`Retry-After` 秒)、その他→`Http`。
- M0 で使うエンドポイント:
  - `GET /2.0/user` — 認証検証(表示名/uuid)
  - `GET /2.0/workspaces?pagelen=50` — 参加ワークスペース一覧(`slug`, `name`)
  - `GET /2.0/repositories/{workspace}?role=member&sort=-updated_on&pagelen=50` — repo一覧(`full_name`, `name`, `updated_on`, `is_private`)

## TUI 仕様

- 端末: raw mode + alternate screen を **RAII ガード**で確保し、**panic hook** で必ず復元(パニック時も端末が壊れない)。
- イベントループ: `tokio::select!` で (a) crossterm `EventStream`、(b) `mpsc::Receiver<Msg>`(API結果) を待つ。API呼び出しは `tokio::spawn` し、結果を `Msg` で返す。
- `Msg` 例: `Key(KeyEvent)`, `Loaded(Loaded)`, `Failed(ApiError)`。
- 画面(`Screen` enum): `Onboarding`, `Workspaces`, `Repositories`, `RepoSelected`。
- 共通キー: `q` / `Ctrl+C` 終了、`Esc` 戻る、`↑/↓` or `j/k` 移動、`Enter` 決定、`?` ヘルプ。
- ローディング中/エラーはステータス行に表示(スピナーは任意)。リスト長い場合はスクロール対応。
- RepoSelected: 選択 repo の `full_name` と「M1: PR一覧をここに実装」を表示するだけ。

## 設定/ログ

- config: `directories::ProjectDirs("dev","","bitbucket-tui")` の config dir に `config.toml`。フィールド: `email`, `display_name`(任意), `default_workspace`(任意)。
- ログ: `BBTUI_LOG` が設定されているときのみ `~/.cache/bitbucket-tui/bitbucket-tui.log` へ tracing 出力。**TUI 実行中に stdout/stderr へ出さない**。

## 受け入れ条件(Acceptance)

- [ ] `rustup run stable cargo build` / `clippy --all-targets -- -D warnings` / `fmt --check` / `test` が全てグリーン。
- [ ] 初回起動 → Onboarding で email+token 入力 → 検証成功 → Workspaces 一覧が出る。
- [ ] 不正トークンで 401 が分かりやすく表示され、再入力できる。
- [ ] Workspace 選択 → Repositories(ページング反映) → repo 選択 → RepoSelected に `full_name` 表示。
- [ ] 再起動時は Keychain の token を使い Onboarding をスキップ。
- [ ] `q` で終了、パニックを起こしても端末が復元される(panic hook 動作)。
- [ ] token が平文ファイルに保存されていない(Keychain のみ)。
- [ ] `bitbucket-tui logout` で認証情報がクリアされる。

## 検証方法

- ユニット: `get_paged` の `next` 追跡、エラー変換(401/403/429)、Paginated のデシリアライズ(モックJSON)。
- 手動: 実 token での起動確認手順を `README.md` に記載(token発行手順とスコープ含む)。
- ネットワークを叩くテストは `#[ignore]` にし、CI ではモックのみ。
