# M7: コメントUIの磨き込みと PR フィルタ

実運用フィードバック第 3 陣。Phase 1（コメントUI磨き + キーヒント）と Phase 2（PR フィルタ）の 2 段で実装する。

## Phase 1

### 1.1 スレッド間の間隔

**問題**: あるスレッドのアクション行/枠下端と、次のスレッドの枠上端（または次の diff 行）が密着していて読みにくい。

**仕様**:
- `CommentRowKind::Spacer` を追加し、**各スレッドブロックの直後**（`push_thread_rows` の `Bottom` の後、`push_collapsed_row` の `Collapsed` の後）に 1 行挟む。
- `Spacer` は non-focusable（`is_focusable` は Header/Collapsed のみ true の現行 match で自動的に false になることを確認）。
- 描画（`comment_box_line`）: 空行（unified/split 両方。split は左カラムも空行）。
- `click_body_row`: `Spacer` クリックは **no-op**（現在の「comment_id 無し → スレッド先頭選択」分岐に落ちないよう、明示的に無視する）。
- ヒットボックス収集（Actions）には影響しない。

### 1.2 コメントエディタのカーソル表示・移動

**問題**: 返信/編集ポップアップで文字入力位置が見えない。

**仕様**:
- `CommentEditor` に `cursor: usize`（`text` の**char 単位**の挿入位置）を追加。`inline`/`reply` は 0（空文字）、`edit` はプリフィル本文の**末尾**で初期化。
- `on_key_comment_editor`:
  - 文字入力/改行（Enter）はカーソル位置に**挿入**、Backspace はカーソル直前を削除（いずれも char 境界で安全に。byte index 演算をしない）。
  - `Left`/`Right`（1 文字）、`Home`/`End`（**論理行**の行頭/行末）、`Up`/`Down`（前/次の論理行へ。列は char 数で維持し行長にクランプ）を追加。
  - 既存の `Ctrl+S` 送信 / `Esc` 取消は不変。
- `render_comment_editor`: カーソル位置の 1 文字を `Style::new().reversed()` で反転表示。カーソルが行末（その行の文字数と同位置）のときは反転させた空白 1 個をその行末に追加（Onboarding の `input_spans`(ui.rs) と同じ視覚方式）。折返し（`Wrap`）は span 単位で維持されるため、反転 span をカーソル char に付けるだけでよい（自前の折返し計算はしない）。
- プレースホルダ（空文字時の「（コメントを入力）」）は反転空白カーソルを先頭に出したうえで維持してよい。

### 1.3 選択ハイライトの可読性

**問題**: 全テーマで `selection_bg == accent == info` のため、選択中コメントの bg 上書きで marker/author/リンク（accent/info の fg）が背景に溶けて読めない。

**仕様**:
- `comment_box_content_line` の hot 時、`highlighted_diff_line` と同じ方式で **全 span の style を `bg(selection_bg) + fg(selection_fg)` に上書き**する（span 個別 fg が Line style より優先されるため、span 単位の上書きが必須）。枠線 span も含めてよい（diff 行ハイライトと同じ見た目に揃える）。
- `Collapsed` の hot 分岐も同様に fg/bg 両上書きへ変更。
- テーマ定義（theme.rs）は**変更しない**。

### 1.4 キーヒントのはみ出し

**問題**: 端末幅が狭いと、フッターのキーヒントが右端で見苦しく切れる。

**仕様**:
- `render_hints` のループで累積幅を追い、**次のエントリが丸ごと収まらなくなったら break**（部分描画しない。省略記号も付けない）。
- ヒットボックス push も同じ break に従う（描画されないエントリはクリック不可のまま）。
- `hint_entries` のデータは変更しない。

### Phase 1 品質ゲート・不変条件

- `cargo fmt --check` / `cargo clippy --all-targets -- -D warnings` / `cargo test --offline` 全 green（現在 543 pass / 1 ignored）。
- 新規依存の追加禁止。テスト外の `unwrap()`/`expect()` 禁止。色は必ず `theme` 経由（状態層 app.rs に `Color::` を持ち込まない）。
- 壊してはならない既存動作: 表示行モデル（`display_rows` の添字 = scroll/cursor）、`rendered_lines`/`rendered_split` の描画キャッシュ、Actions 行のヒットボックス座標、コラプスのトグル/自動コラプス、split の左右対応。
- 新規テスト（最低限）: Spacer が各スレッド後に入り non-focusable であること（display_rows 構成）、エディタのカーソル挿入/移動（Left/Right/Home/End/Up/Down・途中挿入・途中 Backspace）、hot 行の全 span が selection_fg になること、render_hints が狭い幅でエントリ境界で止まること。

## Phase 2（PR フィルタ。Phase 1 完了後に着手）

**ゴール**: PR 一覧で Web と同様に「state の複数選択（例: OPEN+MERGED）」と「author 指定」でサーバサイドにフィルタできる。

### API（事前調査済み。公開リポジトリで実測確認）

- `state` パラメータの繰り返し（`?state=OPEN&state=MERGED`）は OR として有効。値は OPEN/MERGED/DECLINED/SUPERSEDED。
- `q=`（BBQL）: `state IN ("OPEN","MERGED")`・`author.uuid="{uuid}"`・AND 連結・`sort` 併用すべて有効。
- **重要（実測）: `q` と `state` パラメータを同時指定すると `state` 側が黙って無視される**。混在させないこと。
- `q` の値は全体を URL エンコードする（reqwest のクエリ機構に委ねる）。
- クエリ構築規則: **author 指定なし → 従来どおり `state` 繰り返し**（実績のある経路を維持）。**author 指定あり → `q=state IN (...) AND author.uuid="{uuid}"` に一本化し `state` パラメータは付けない**。いずれも `sort` は併用。

### フィルタ状態

- `PrStateFilter`（enum）を「states の集合 + author」のフィルタ構造体へ拡張（例: `states: BTreeSet<PrState>`、`author: Option<PrAuthor{uuid, display_name}>`。Hash/Eq 可能に）。
- 既存の `o`/`m`/`d`/`a` 単発キーは「その state のみ（All は 4 つ全部）」の集合へ写像して維持。
- `PullRequestsCache` のキーへ author（uuid）と states 集合を含める（キー衝突による誤表示を防ぐ）。
- 永続化: **states の選択は config.toml に保存**（文字列配列）。author はセッション内のみ（リポジトリ依存のため保存しない）。

### UI

- PR 一覧で `f` を押すとフィルタモーダル（`f`/`F` は未割当を確認済み。`a` は既存 All で使用中のため使わない）。
- モーダル構成: 上段 **State**（4 つのチェックボックス。`↑↓/jk` 移動・`Space` トグル）、下段 **Author**（`All authors` + 候補リスト。`↑↓/jk` 選択）。`Tab` でセクション移動、`Enter` で適用（1 ページ目から再取得）、`Esc` で取消。states が 0 個で Enter したら Status エラー（適用しない）。
- author 候補: **ワークスペースメンバー API**（`GET /2.0/workspaces/{workspace}/members`、応答は `values[].user`。ページング追従）から**モーダル初回表示時に遅延取得しワークスペース単位でキャッシュ**。取得失敗（権限等）時は**読み込み済み PR の author から重複排除して**候補にする（フォールバック。uuid 無しの author は候補から除外）。
- フィルタ適用中はタイトル等にフィルタ内容を表示（例: ` PR (OPEN+MERGED, author: Alice) `）。

### Phase 2 品質ゲート・不変条件

- Phase 1 と同じゲート・不変条件。
- 追加: members API の応答形（`values[].user` の uuid/display_name）・`q` の実挙動は「未検証の仮定」として LEDGER に記録する（読み取り系なので実 token での検証はユーザーに委ねる）。
- 新規テスト: クエリ構築（author 有無での `q`/`state` 切替、URL エンコードは reqwest 委譲の確認）、フィルタ構造体のキャッシュキー、モーダルのキー操作（トグル・適用・空集合拒否）、単発キー写像、config 永続化 round-trip。

# M8: 一覧・Diff の磨き込み第 2 陣（M7 と同ファイルに追記）

実運用フィードバック第 4 陣。Phase 1（調査不要分）と Phase 2（API 調査待ち: target branch フィルタ）に分ける。

## Phase 1

### 8.1 キーヒントの方針刷新
- **全画面**の `hint_entries` から `PgUp/PgDn`（`f`/`b` 表記含む）と `Shift+J/K` のエントリを削除（機能は残す。ヘルプ `?` の全文には残してよい）。
- **`?` は必ず表示**: 各画面の `hint_entries` の**先頭**に `("?", "ヘルプ")` を置き、幅切詰めで絶対に落ちないことをテストで固定（現状 `?` がどこで付与されているかを確認し、二重表示にならないよう整理）。
- 並び順の方針: `?` → **画面固有のアクションキー** → 汎用（`Enter`/`Esc`/`↑↓jk` 等）。幅が足りないときは末尾（汎用側）から落ちる。
- **PR 一覧**: `o`/`m`/`d`/`a` はヒントに出さない（機能とヘルプは残す）。フィルタ系は `f` のみ表示。

### 8.2 吹き出しアイコンを 💬 に戻す
- コメントヘッダの marker とサイドバーのコメント数バッジを 🗨 から **💬**（元の絵文字）へ戻す。幅計算は従来どおり UnicodeWidthStr 経由（💬 は幅 2 として整合していた実績あり）。関連テストの期待値を更新。

### 8.3 unified（1 枚表示）の行番号ガター
- Diff の unified 本文の各行の左に、**旧行番号列・新行番号列の 2 列**のガターを表示する（参照スクリーンショット準拠。削除行=旧のみ、追加行=新のみ、文脈行=両方、FileHeader/Hunk/Meta 行は空白）。
- 実装: `rendered_lines` 構築時に各行へ `{old:>ow$} {new:>nw$} ` のプレフィックス span（`theme.muted`）を焼き込む（ow/nw はその diff 全体の最大行番号の桁数。キャッシュ機構は不変）。
- カーソルハイライト・コメントボックス（全幅のまま）・アクションリンクのヒットボックス・`comment_anchor`・位置ラベルはいずれも影響を受けないこと（テキスト先頭にガター span が増えるのみ）。
- split 表示は対象外（従来どおり）。
- テスト: 行番号が描画されること、桁揃え（最大行番号の桁数）、メタ/ハンク行は空白ガター。

### 8.4 フィルタモーダルの author 検索
- Author セクションに**そのまま入力で絞り込み**（パレット式）を追加: 印字文字で検索クエリに追記、`Backspace` で削除、候補は fuzzy（既存の nucleo 経路を流用）で絞り込み、`↑↓` で選択。**Author セクションでは `j`/`k` は検索文字として扱う**（移動は `↑↓`）。State セクションの操作（`jk` 移動・`Space` トグル）は従来どおり。
- クエリはセクション移動（Tab）では保持、モーダルを閉じたら破棄。候補 0 件時は「(該当なし)」を表示し Enter は無効（現用 author 維持）。
- 検索クエリ表示行をモーダルに追加（例: `Author: alic▏`）。

### 8.5 ✔ バッジは 0/0 のとき非表示
- PR 一覧の `✔n/m`（承認済み/レビュアー総数）バッジは **`reviewer_count() == 0` のとき出さない**。詳細画面の表示は現状維持。テストで固定。

### Phase 1 品質ゲート・不変条件
- `fmt --check` / `clippy --all-targets -- -D warnings` / `test --offline` 全 green（**現在 584 passed / 1 ignored**）。
- 新規依存禁止・テスト外 unwrap/expect 禁止・状態層に Color 禁止。
- 壊さない: 表示行モデル、描画キャッシュ、アクションリンクのヒットボックス、コラプス、split 表示、M7 のフィルタ動作（states/author/永続化）。

## Phase 2（API 調査完了後に確定）
- フィルタモーダルに **Target branch** セクションを追加（そのまま入力で絞り込み。候補はブランチ一覧 API またはサーバサイド部分一致——調査結果で確定）。サーバサイドの BBQL（`destination.branch.name`）でフィルタ。
- author 候補の「repo 権限者への限定」可否は調査結果を踏まえて回答・設計。

## M8 Phase 2（仕様確定版。Phase 1 完了後に着手）

### API（事前調査済み。公開リポジトリで実測確認）
- `q=destination.branch.name ~ "substr"` は**大文字小文字を区別しない部分一致**として有効。`=` は完全一致。`author.uuid` との AND 連結も実測済み。
- `author.display_name` は**フィルタ不可（400）**。author 条件は uuid（現行どおり）を使う。
- ブランチ一覧 `GET /2.0/repositories/{ws}/{repo}/refs/branches` は `q=name ~ "..."` のサーバサイド部分一致に対応（今回は候補の初回 1 ページ取得＋ローカル絞り込みで十分。サーバサイド絞り込みは将来拡張）。
- 権限保有者一覧 API は管理者限定のため、author 候補の「repo 権限者への限定」は**採用しない**（現行の members + フォールバックを維持）。

### 8.6 Target branch フィルタ
- フィルタ構造体に `target_branch: Option<TargetBranch>` を追加。`TargetBranch { text: String, exact: bool }`（Hash/Eq。キャッシュキーに含める）。
  - `exact=true`: 候補（実在ブランチ）から選択 → `destination.branch.name = "name"`。
  - `exact=false`: 自由入力 → `destination.branch.name ~ "text"`（部分一致）。
- クエリ構築: author か target_branch の**どちらかが有効なら `q=` 一本化**（`state IN (...) [AND author.uuid="..."] [AND destination.branch.name =|~ "..."]`。`state` パラメータは付けない）。どちらも無しは従来の `state` 繰り返し。
- モーダルに **Target branch セクション**（3 番目）を追加: パレット式入力（Author と同じ操作系。印字文字で入力・Backspace・`↑↓` で候補移動）。候補リストは先頭に「All branches」、クエリ非空なら次に「『{query}』で部分一致」（exact=false の適用行）、続いて実在ブランチ候補（fuzzy 絞り込み。選択すると exact=true）。
- ブランチ候補: モーダル初回表示時に既存のブランチ一覧取得（1 ページ目のみ、既存の client 関数流用）を遅延実行し repo 単位でキャッシュ。取得失敗時は候補なし（自由入力の部分一致は使える）。
- author と同様に**セッション内のみ**（永続化しない）。repo コンテキストが変わったら author と一緒にリセット。一覧タイトルへ表示（例: ` PR [OPEN, target~"release"] ` / 完全一致は ` target: main `）。
- キャッシュキー・文脈ガードに target_branch を含める。
- テスト: クエリ構築（`~`/`=` の切替・AND 連結・両方無しの従来経路）、キャッシュキー分離、モーダル操作（入力→部分一致適用・候補選択→完全一致適用・All branches でクリア）、repo 切替リセット、タイトル表示。

### 8.7 author 候補を「この repo の PR 著者」に変更（ユーザー決定）
- 候補ソースを**ワークスペースメンバー API から「この repo の PR 著者の集約」へ変更**する（権限者一覧 API は管理者限定で使えないため、"この repo に関わる人" の実用的近似。メンバー大量問題も解消）。
- 取得: フィルタモーダル初回表示時に、PR 一覧 API を **全 4 state の `state` 繰り返し・`pagelen=50`・最新順で最大 3 ページ**（`next` が残っていても打切り）遅延取得し、`author` を uuid で重複排除・表示名昇順で候補化。**repo 単位で成功時のみキャッシュ**。
- 取得失敗時: 既存の `PullRequestsCache` 集約（現行 `loaded_pr_authors`）へフォールバック。
- 現用 author が候補に無ければ挿入・読み込み中 Enter は現用 author 維持（M7/レビュー反映の規約を維持）。
- **members API 関連コードは削除**する（`get_workspace_members` / `WorkspaceMember` / `workspace_members_cache` / `LoadWorkspaceMembers` / `WorkspaceMembersLoaded` と関連テスト。未使用コードを残さない）。
- 既知の性質として LEDGER に記録: 候補は「直近 150 件の PR の著者」なので、それより古い PR にしか登場しない著者は候補に出ない（自由入力は無いが、author フィルタの目的上、直近に PR が無い著者で絞る需要は低い想定）。
