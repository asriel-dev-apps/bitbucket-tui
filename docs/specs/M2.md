# M2 実装仕様書 — パイプライン監視

対象: `bitbucket-tui/`。上位方針は `../design.md`、規約は `../../CLAUDE.md`、環境事実・M0/M1実装メモは `../LEDGER.md` を必ず読むこと。M0/M1 の上に載せる。

## ゴール

選択リポジトリの **Bitbucket Pipelines を監視**できるようにする: パイプライン一覧(状態つき) → 詳細(ステップ一覧) → ステップログ閲覧、そして re-run / stop。CI の状況をキーボードで素早く追える監視ビューを提供する。

## スコープ(M2でやること)

1. API クライアント拡張(pipelines 系エンドポイント + ステップログのテキスト取得)。
2. 画面追加: `Pipelines`(一覧) → `PipelineDetail`(ステップ一覧) → `StepLog`(ログ閲覧, スクロール)。
3. アクション: **re-run**(同一 target で trigger) / **stop**(stopPipeline)。破壊的操作は確認モーダル。
4. ナビゲーション: repo コンテキストから Pipelines に入れるようにする。
5. 進行中パイプラインの **状態ポーリング更新**(手動リフレッシュ + 任意で自動リフレッシュ)。

### やらないこと(後回し)

- パイプラインの詳細な変数入力つきカスタム trigger(re-run は元 target の再実行のみ)。M3(ブランチ/ソース閲覧)。
- ログの真のライブストリーミング(API は静的テキスト+Range。進行中は再取得=擬似 tail で対応)。
- ステップ単位の再実行、キャッシュ/アーティファクト操作。

## 依存(重要)

**新規クレート追加は禁止**(依存凍結・ネットワーク不可)。既存依存のみ。ログビューアは M1 の diff/pager 実装を流用または汎用スクロールビューで。新クレートが要ると判断したら追加せず報告して止まること。

## API 仕様(追加)

base = `https://api.bitbucket.org/2.0`。`{ws}`/`{repo}` は slug。**`{pipeline_uuid}`・`{step_uuid}` は波括弧 `{...}` を含む UUID で、URL に入れる際は波括弧ごと percent-encode すること**(素の UUID だと `The value provided is not a valid uuid` になる。既知の罠)。認証・ページング・エラー変換・生テキスト取得は M0/M1 の `api` を再利用。

- 一覧: `GET /repositories/{ws}/{repo}/pipelines?sort=-created_on&pagelen=50`
  - フィルタは `q`(BBQL, 例 `target.ref_name="main"` や `status`)。既定は新しい順。
- 詳細: `GET /repositories/{ws}/{repo}/pipelines/{pipeline_uuid}`
- ステップ一覧: `GET /repositories/{ws}/{repo}/pipelines/{pipeline_uuid}/steps/`(ページング)
- ステップ: `GET /repositories/{ws}/{repo}/pipelines/{pipeline_uuid}/steps/{step_uuid}`
- ステップログ(テキスト): `GET /repositories/{ws}/{repo}/pipelines/{pipeline_uuid}/steps/{step_uuid}/log` → **`text/plain`(大きい場合あり。HTTP Range 対応。まずは全取得で可、巨大時に末尾 N バイトを Range 取得する実装は任意)**。ログが未生成(未開始/生成前)だと 404 になり得る → 「ログなし」を表示。
- stop: `POST /repositories/{ws}/{repo}/pipelines/{pipeline_uuid}/stopPipeline`(未完了ステップを停止)。
- re-run(trigger): `POST /repositories/{ws}/{repo}/pipelines/`、body 例 `{"target":{"type":"pipeline_ref_target","ref_type":"branch","ref_name":"<branch>","selector":{"type":"default"}}}`。元パイプラインの `target` を引き継いで再実行する。

> **モデル(serde)**: `Pipeline`{uuid,build_number,state{name,result?{name},stage?{name}},creator?(User),created_on,completed_on?,target{ref_type?,ref_name?,commit?{hash},selector?},trigger?{name},duration_in_seconds?}、`PipelineStep`{uuid,name?,state{name,result?{name}},started_on?,completed_on?,duration_in_seconds?}、`PipelineTarget`(re-run 用)。
> 状態値(要実データ検証): pipeline `state.name` = `PENDING`|`BUILDING`|`IN_PROGRESS`|`COMPLETED`|`STOPPED`|... 、`result.name` = `SUCCESSFUL`|`FAILED`|`STOPPED`|`ERROR`|... 。step も同様。**フィールドは `Option`/`#[serde(default)]` で耐性を持たせ、実 API 応答で検証**して LEDGER に反映。
> 状態→色: 成功=緑, 失敗/エラー=赤, 進行中=黄, 停止/中止=グレー, 保留=既定。

必要スコープ: `pipeline`(読取), `pipeline:write`(stop/trigger)。

## TUI 仕様(追加)

- `Screen` に `Pipelines`, `PipelineDetail`, `StepLog` を追加。
- **ナビゲーション**: Pipelines は repo スコープ。`Repositories` で repo に対し `p`=Pipelines(既存の `Enter`=PullRequests はそのまま)。`PullRequests` 画面で `P`=同じ repo の Pipelines に切替。`Pipelines` から `Esc`=`Repositories`(または直前画面)へ。相互に行き来しやすくする。
- **Pipelines(一覧)**: 各行 = `#build_number  状態(色)  target(ref_name)  trigger  作成日時  所要`。`Enter`=詳細、`r`=再読込、`S`=stop(進行中のみ・確認モーダル)、`R`=re-run(確認モーダル)。ページング対応。
- **PipelineDetail**: 上部にメタ(build#/状態/target/creator/作成〜完了/所要)、下部にステップ一覧(各ステップの状態色・名前・所要)。`Enter`=選択ステップの StepLog、`S`=stop、`R`=re-run、`Esc`=一覧。
- **StepLog**: ログをスクロール表示(`↑/↓ j/k`・`PgUp/PgDn`・`g/G`)。`r`=再取得(進行中の擬似 tail)。ログ未生成は「(ログなし)」。`Esc`=詳細。ANSI エスケープは可能なら簡易処理(最低限そのまま表示でも可、制御文字は除去)。
- **確認モーダル**: `stop`・`re-run` は必ず確認(M1 の merge モーダルの仕組みを流用)。実行後はパイプライン状態を再取得して反映。
- **状態ポーリング(監視の肝)**: 進行中(PENDING/IN_PROGRESS/BUILDING)のパイプラインがある一覧・詳細では、**一定間隔(例 5s)で自動リフレッシュ**する仕組みを入れる(tokio のタイマ tick を `Msg` として流す)。全て完了状態なら自動リフレッシュは止める。手動 `r` は常に可。自動リフレッシュの ON/OFF をキー(例 `a`)で切替できると良い。
- 共通キー(踏襲): `q`/`Ctrl+C`終了, `Esc`戻る, `?`ヘルプ(M2 キー追記)。非同期は M0/M1 同様 `tokio::spawn`→`Msg`。ローディング/エラーはステータス行。

## モジュール変更(目安)

- `api/models.rs`: 上記モデル追加。`api/client.rs`: `list_pipelines`/`get_pipeline`/`list_pipeline_steps`/`get_pipeline_step`/`get_step_log`(text)/`stop_pipeline`/`trigger_pipeline`。**uuid の波括弧込み percent-encode ヘルパ**を追加。
- `tui/app.rs`: `Screen`/`Msg`/`Command` 拡張、ポーリング用の tick 処理。`tui/ui.rs`: 新画面描画 + 状態色 + ログビューア。`tui/event.rs`: 新 Command dispatch + タイマ tick。必要なら `tui/logview.rs`。
- `docs/design.md` の画面遷移を更新。

## 品質バー(M0/M1 と同じ、全て green)

- `export PATH="$HOME/.rustup/toolchains/stable-aarch64-apple-darwin/bin:$PATH"` の上で
  `cargo fmt --check` / `cargo clippy --all-targets --offline -- -D warnings` / `cargo test --offline`。
- `unwrap`/`expect` はテスト以外禁止。bin=anyhow, api=thiserror。
- ネットワークを叩くテストは `#[ignore]`。通常テストはモック: Pipeline/Step デシリアライズ、uuid エンコード(波括弧)、trigger body 生成、状態→色マップ、`update()` の遷移とアクション発行、ポーリング tick でリフレッシュが発行されること。
- TUI 中 stdout/stderr 禁止。端末は既存 `Tui` 再利用。破壊的操作(stop/re-run)は確認モーダル経由。crossterm 型は `ratatui::crossterm` 経由。

## 受け入れ条件(Acceptance)

- [ ] fmt/clippy(-D warnings)/test すべて green(offline)。
- [ ] repo から Pipelines 一覧に入れる(状態が色付き)。
- [ ] パイプライン選択 → 詳細(ステップ一覧)表示。
- [ ] ステップ選択 → ログをスクロール表示(未生成は「ログなし」)。
- [ ] 進行中パイプラインがあると一覧/詳細が自動リフレッシュ(手動 `r` も可)。全完了で自動停止。
- [ ] stop・re-run が確認モーダル経由で実行でき、実行後に状態が更新。破壊的操作は確認なしに走らない。
- [ ] uuid は波括弧込みで正しくエンコードされ、404/エラーがステータス行に表示。

## 検証方法

- ユニット: 上記モック検証。ネット依存は `#[ignore]`。
- 手動(実 token 必要・任意): 進行中パイプラインで一覧自動更新→ログ閲覧→stop→re-run を一巡。手順を README に追記。
- 実 API で判明した状態値・result 値・ログ 404 挙動・trigger body の正確形を `docs/LEDGER.md` に反映。
